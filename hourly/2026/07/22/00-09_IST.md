## Mental-model update

| Previous assumption | Updated model | Architecture consequence |
|---|---|---|
| Choose one best model for an agent | Route tasks among models based on difficulty, latency, tool-use reliability, context, and cost | Build explicit routing policy, fallback, telemetry, and per-route evaluation rather than embedding one model name throughout the system |
| Token price predicts agent cost | Agent cost depends on output tokens, reasoning steps, tool calls, retries, latency, and human rework | Measure cost per accepted task and cost per quality-gated change, not price per million tokens alone |
| Small/fast models are only for trivial chat | Throughput models can execute bounded subagent, extraction, classification, and document-processing work | Decompose workflows into verifiable stages and reserve expensive models for uncertain planning and synthesis |
| Strong cyber capability is just a model property | Cyber capability emerges from specialized training plus orchestration, validation, containment, and access policy | Evaluate the complete cyber-agent system and apply staged deployment, identity, logging, and human approval |

## Architecture implications

| Decision area | Recommended engineering response |
|---|---|
| Model routing | Use a task taxonomy and route by complexity, latency SLO, multimodal need, tool requirement, risk, and expected cost |
| Evaluation | Compare complete trajectories: completion rate, unwanted edits, tool calls, retries, wall-clock latency, accepted output, rework, and spend |
| Reliability | Add fallbacks, confidence/uncertainty triggers, deterministic validators, and route-specific rollback |
| Governance | Version model IDs, routing rules, prompts, tools, and evaluation sets; retain provider-independent traces |
| Security | Treat built-in computer use and code execution as privileged capabilities with sandboxing, scoped identity, egress controls, and trajectory monitoring |
