## Mental-model update

| Previous assumption | Updated mental model | Architecture consequence |
|---|---|---|
| MCP is a stateful conversational transport | MCP should be treated as a stateless tool-RPC protocol; workflow state belongs in explicit application handles | Separate transport state, workflow state, identity state, and long-running task state |
| SDK compatibility is enough to validate a migration | SDK compatibility is necessary but insufficient; conformance, integration, authorization, and workload replay are separate gates | Add protocol-version matrices and executable conformance to CI |
| Gateways must inspect JSON-RPC bodies to route and govern tools | Stable operation metadata can support routing and policy before payload parsing | Put method/name-aware controls at the gateway while retaining payload validation downstream |
| Statelessness removes observability context | Trace context and explicit request metadata can improve cross-service correlation | Standardize OpenTelemetry propagation across host, MCP client, server, and downstream tools |
| Long-running tasks belong in hidden sessions | Tasks should expose explicit handles and lifecycle operations | Make ownership, cancellation, expiry, authorization, and replay testable |

## Architecture implication

```text
agent host
  → protocol-version negotiation
  → stateless MCP request
  → method/name-aware gateway policy
  → any healthy MCP server instance
  → explicit workflow or task handle
  → downstream tool/service
  → trace-correlated result
  → conformance + integration + authorization evidence
```

## Terms to avoid misusing

| Term | Misuse | Correct interpretation |
|---|---|---|
| Stateless | “The agent or workflow has no state.” | The protocol core carries no hidden session; applications may use explicit state handles |
| Backward compatible | “No migration testing is required.” | Tier-1 SDKs may preserve compatibility while application-level behaviors and deprecated features still require validation |
| Conformant | “The implementation is production ready.” | It passes protocol scenarios; security, reliability, latency, scale, and business correctness remain separate |
| MCP-native | “Uses MCP somewhere.” | Implements a declared specification version, lifecycle, authorization, observability, and conformance contract |
